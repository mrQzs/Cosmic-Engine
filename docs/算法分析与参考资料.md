## 三、各算法参考资料索引

### 3.1 GLSL 着色器与图形学

#### 3.1.1 Voronoi Noise (泰森多边形)

- **核心概念**: 基于随机种子点计算每个像素到最近种子点的距离，形成细胞状分区图案
- **参考资料**:
  - [The Book of Shaders - Chapter 12: Cellular Noise](https://thebookofshaders.com/12/) -- 最佳入门教程，含可交互编辑器
  - [Inigo Quilez - Voronoi Edges](https://iquilezles.org/articles/voronoilines/) -- Shader 大师 iq 的 Voronoi 边缘计算
  - [Inigo Quilez - Smooth Voronoi](https://iquilezles.org/articles/smoothvoronoi/) -- 平滑 Voronoi 变体
  - [Shadertoy 示例: Voronoi - basic](https://www.shadertoy.com/view/MslGD8) -- 可直接运行的 GLSL 代码
  - **论文**: Worley, S. "A Cellular Texture Basis Function" (SIGGRAPH 1996)

#### 3.1.2 Simplex / Perlin Noise

- **核心概念**: 梯度噪声函数，用于程序化生成自然纹理（地形、云层、岩石）
- **参考资料**:
  - [The Book of Shaders - Chapter 11: Noise](https://thebookofshaders.com/11/) -- 从零理解噪声函数
  - Stefan Gustavson - "Simplex noise demystified" (2005) -- Simplex Noise 论文级解析
  - [GPU Gems 2 - Chapter 26: Implementing Improved Perlin Noise](https://developer.nvidia.com/gpugems/gpugems2/part-iii-high-quality-rendering/chapter-26-implementing-improved-perlin-noise)
  - [lygia GLSL 噪声库](https://github.com/patriciogonzalezvivo/lygia) -- 可直接 `#include` 的 GLSL 噪声函数集
  - [webgl-noise](https://github.com/stegu/webgl-noise) -- Stefan Gustavson 的 WebGL 噪声 GLSL 实现

#### 3.1.3 Raymarching (光线步进)

- **核心概念**: 沿视线方向以步进方式逼近隐式曲面（SDF），适合渲染无显式几何的效果（黑洞、云雾）
- **参考资料**:
  - [Inigo Quilez - Distance Functions](https://iquilezles.org/articles/distfunctions/) -- SDF 函数百科全书
  - [Inigo Quilez - Raymarching Primitives](https://iquilezles.org/articles/raymarchingdf/) -- Raymarching 基础教程
  - [Jamie Wong - Ray Marching and Signed Distance Functions](http://jamie-wong.com/2016/07/15/ray-marching-signed-distance-functions/)
  - [Shadertoy 示例: Black Hole Raymarching](https://www.shadertoy.com/view/tsBXW3)
  - [The Art of Code YouTube](https://www.youtube.com/c/TheArtofCodeIsCool) -- Raymarching 视频教程系列

#### 3.1.4 施瓦西度规 / 引力透镜

- **核心概念**: 广义相对论中静态球对称黑洞周围的时空度规，用于计算光线在引力场中的弯曲路径
- **参考资料**:
  - [Shadertoy: Black Hole Gravitational Lensing](https://www.shadertoy.com/view/Xdf3zN) -- 经典黑洞着色器实现
  - Riazuelo, A. "Seeing relativity—I. Ray tracing in a Schwarzschild metric" (2019) -- 学术级可视化
  - [YouTube: Sebastian Lague - Black Holes (Coding Adventure)](https://www.youtube.com/watch?v=Iaz9TqYWUmA) -- 极佳的编程实践视频
  - **简化公式**: `deflection_angle ≈ 4GM/(rc²)` 在 Shader 中通过 UV 偏移 `uv += normalize(uv) * strength / length(uv)` 近似实现

#### 3.1.5 Fresnel Effect (菲涅尔效果)

- **核心概念**: 视线与表面法线夹角越大，反射越强——用于星球大气层边缘发光
- **参考资料**:
  - [Three.js 示例: Fresnel Shader](https://threejs.org/examples/#webgl_materials_envmaps) -- 官方示例
  - [Codrops: Fresnel Effect in GLSL](https://tympanus.net/codrops/2024/01/02/creating-a-fresnel-effect-with-glsl/) -- 含完整代码
  - **GLSL 核心代码**: `float fresnel = pow(1.0 - dot(viewDir, normal), power);`

#### 3.1.6 Smoothstep 与混合函数

- **核心概念**: 平滑插值函数 `3t² - 2t³`，用于纹理渐变、距离混合
- **参考资料**:
  - [The Book of Shaders - Chapter 5: Shaping Functions](https://thebookofshaders.com/05/)
  - [Inigo Quilez - Smooth Minimum/Maximum](https://iquilezles.org/articles/smin/)

### 3.3 空间分布与天体运动算法

#### 3.3.1 Fibonacci Sphere (斐波那契球面均匀分布)

- **核心概念**: 在球面上近似均匀分布 N 个点，用于卫星防穿模轨道分配
- **参考资料**:
  - [Extreme Learning - Fibonacci Sphere](https://extremelearning.com.au/how-to-evenly-distribute-points-on-a-sphere-more-effectively-than-the-canonical-fibonacci-lattice/) -- 最佳讲解文章
  - González, Á. "Measurement of Areas on a Sphere Using Fibonacci and Latitude–Longitude Lattices" (2010) -- 数学证明
  - **Go 伪代码**:
    ```
    golden_ratio = (1 + sqrt(5)) / 2
    for i in 0..N:
        θ = acos(1 - 2*(i+0.5)/N)
        φ = 2π * i / golden_ratio
        x, y, z = sin(θ)*cos(φ), sin(θ)*sin(φ), cos(θ)
    ```

#### 3.3.2 轨道力学 (简化开普勒运动)

- **核心概念**: 天体沿椭圆/圆形轨道公转，使用参数方程 + 时间偏移
- **参考资料**:
  - [Three.js Journey - Particles](https://threejs-journey.com/) -- Bruno Simon 课程中的粒子轨道动画
  - **核心公式**:
    ```js
    x = r * Math.cos(time * speed + phaseOffset)
    y = r * Math.sin(orbit_inclination) * Math.sin(time * speed + phaseOffset)
    z = r * Math.sin(time * speed + phaseOffset)
    ```

#### 3.3.3 CatmullRom Spline (Catmull-Rom 样条曲线)

- **核心概念**: 通过控制点生成平滑曲线，自动计算切线，适合路径动画
- **参考资料**:
  - [Three.js 文档: CatmullRomCurve3](https://threejs.org/docs/#api/en/extras/curves/CatmullRomCurve3)
  - [Catmull-Rom Splines 可视化](https://www.mvps.org/directx/articles/catmull/) -- 交互式理解

### 3.4 空间数据结构

#### 3.4.1 Octree (八叉树)

- **核心概念**: 将 3D 空间递归划分为 8 个子立方体，加速空间查询（射线检测、邻近搜索）
- **参考资料**:
  - [Learn OpenGL - Octree](https://learnopengl.com/Guest-Articles/2021/Scene/Frustum-Culling) -- 结合视锥剔除

  - [Real-Time Rendering, 4th Edition](https://www.realtimerendering.com/) -- 第 19 章 加速结构
  - [GitHub: sparse-octree](https://github.com/vanruesc/sparse-octree) -- Three.js 生态的稀疏八叉树实现

#### 3.4.2 Frustum Culling (视锥剔除)

### 3.6 后端算法

#### 3.6.1 Token Bucket (令牌桶限流)

- **核心概念**: 固定速率向桶中添加令牌，请求需消耗令牌；桶空则拒绝
- **参考资料**:
  - [Go 标准库: golang.org/x/time/rate](https://pkg.go.dev/golang.org/x/time/rate) -- Go 官方令牌桶实现
  - [System Design Primer - Rate Limiting](https://github.com/donnemartin/system-design-primer#rate-limiting) -- 系统设计角度
  - Fiber 中间件: `fiber.middleware.limiter` 内置令牌桶

#### 3.6.2 HyperLogLog (基数估计)

- **核心概念**: 用极少内存（~12KB）估算百万级集合的不同元素数量，误差 ~0.81%
- **参考资料**:
  - [Redis 文档: PFADD / PFCOUNT](https://redis.io/docs/latest/commands/pfadd/) -- Redis HyperLogLog 命令
  - Flajolet, P. et al. "HyperLogLog: the analysis of a near-optimal cardinality estimation algorithm" (2007) -- 原始论文

#### 3.6.3 Exponential Backoff (指数退避)

- **核心概念**: 失败后等待时间指数增长（1s, 2s, 4s, 8s...），避免雪崩
- **参考资料**:
  - [AWS Architecture Blog - Exponential Backoff And Jitter](https://aws.amazon.com/blogs/architecture/exponential-backoff-and-jitter/)
  - Go 实现: `cenkalti/backoff` 库

#### 3.6.4 DataLoader / N+1 查询优化

- **核心概念**: 批量化 + 缓存数据库请求，解决 GraphQL 嵌套查询的 N+1 问题
- **参考资料**:
  - [graph-gophers/dataloader](https://github.com/graph-gophers/dataloader) -- Go DataLoader 库
  - [dataloaden](https://github.com/vektah/dataloaden) -- gqlgen 推荐的代码生成 DataLoader
  - [gqlgen 文档: Dataloaders](https://gqlgen.com/reference/dataloaders/)

#### 5.2.1 Lerp / Slerp (线性插值 / 球面线性插值)

- **重要程度**: **极高** (所有平滑运动的基础)
- **缺口说明**: PLAN 大量提到"平滑移动"、"平滑过渡"，但从未提及底层插值算法。GSAP 内部使用 Lerp，但自定义 `useFrame` 动画（如在线用户飞船位置同步、卫星轨道重排）必须手写。
- **使用场景**:
  - 摄像机飞行到目标位置的逐帧逼近
  - 在线用户飞船位置的网络同步平滑（PLAN_06 §11.4 提到"平滑插值"但未展开）
  - 卫星排序切换时的位置过渡
  - 乐观更新中临时卫星→正式卫星的轨道迁移
  - LOD 切换时材质/大小的渐变
- **核心公式**:
  ```js
  // Lerp: 每帧从当前值向目标值靠近一定比例
  current = current + (target - current) * smoothFactor * deltaTime

  // Slerp: 球面线性插值（用于旋转/方向）
  THREE.Quaternion.slerp(quatA, quatB, resultQuat, t)
  ```
- **参考资料**:
  - [Three.js MathUtils.lerp](https://threejs.org/docs/#api/en/math/MathUtils.lerp)
  - [Three.js Vector3.lerp / Quaternion.slerp](https://threejs.org/docs/#api/en/math/Vector3.lerp)
  - [Frame Rate Independent Lerp Explained](https://www.gamedeveloper.com/programming/improved-lerp-smoothing-) — 为什么 `lerp(a, b, 0.1)` 在不同帧率下表现不同及修复方法

---

#### 5.2.2 Delta Time (帧率无关动画)

- **重要程度**: **极高** (不用 = 高帧率设备动画翻倍快)
- **缺口说明**: PLAN 的所有 `useFrame` 动画代码示例（如 `x = r * cos(time + offset)`）均使用 R3F 提供的 `clock.elapsedTime`，这本身没问题。但对于**增量式动画**（如粒子移动、自转速度），如果不乘以 `delta`，在 30fps 和 144fps 设备上动画速度会差 4.8 倍。
- **核心模式**:
  ```js
  useFrame((state, delta) => {
    // 错误: 高帧率设备旋转更快
    mesh.rotation.y += 0.01

    // 正确: 帧率无关
    mesh.rotation.y += rotationSpeed * delta
  })
  ```
- **参考资料**:
  - [R3F useFrame 文档](https://docs.pmnd.rs/react-three-fiber/api/hooks#useframe) — delta 参数说明
  - [Fix Your Timestep!](https://gafferongames.com/post/fix_your_timestep/) — Glenn Fiedler 经典文章

---

#### 5.2.3 Object Pooling (对象池)

- **重要程度**: **高** (粒子系统、评论卫星频繁创建/销毁)
- **缺口说明**: 文档中完全未提及。以下场景会频繁创建和销毁 3D 对象，触发 GC 卡顿：
  - 卫星发射动画中的粒子 Trail
  - 引力坍缩爆炸的碎片粒子
  - 虫洞跃迁的星线拉丝粒子
  - 搜索高亮时天体的临时高亮效果
- **核心思路**: 预创建固定数量的对象，使用时激活 + 设置位置，用完归还池而非销毁
- **参考资料**:
  - [Three.js 粒子池示例](https://github.com/mrdoob/three.js/blob/master/examples/webgl_points_dynamic.html) — 动态粒子的池化方案
  - [drei `<Instances>`](https://github.com/pmndrs/drei#instances) — 声明式实例管理（部分替代对象池）
  - [Object Pool Pattern (Game Programming Patterns)](https://gameprogrammingpatterns.com/object-pool.html) — Robert Nystrom 经典讲解

---

#### 5.2.4 GPU 内存管理 / dispose() 模式

- **重要程度**: **高** (不处理 = 必然内存泄漏)
- **缺口说明**: Three.js 的 Geometry、Material、Texture 不会被 JS GC 自动回收 GPU 内存，必须手动 `dispose()`。PLAN 中**零处提及**。以下场景是泄漏高发区：
  - LOD 切换时旧 Mesh 的 geometry/material
  - 虫洞跃迁离开旧星区后，旧星区天体的纹理
  - 编辑器中预览图片的 Texture
  - Shader Material 的 uniform texture
- **核心模式**:
  ```js
  useEffect(() => {
    return () => {
      geometry.dispose()
      material.dispose()
      texture.dispose()
    }
  }, [])
  ```
- **参考资料**:
  - [Three.js 文档: How to dispose of objects](https://threejs.org/docs/#manual/en/introduction/How-to-dispose-of-objects) — 官方指南（必读）
  - [R3F 自动 dispose](https://docs.pmnd.rs/react-three-fiber/api/objects#disposal) — R3F 组件卸载时自动 dispose（但自定义资源需手动处理）
  - [detect-gpu](https://github.com/pmndrs/detect-gpu) — 检测 GPU 等级，决定纹理质量上限

---

#### 5.2.5 BVH Raycasting (层级包围体加速射线检测)

- **重要程度**: **高** (天体数量多时点击/悬停检测会卡)
- **缺口说明**: PLAN §4.6 提到天体点击交互使用 R3F 事件系统，但 R3F 底层用的是 Three.js 的 `Raycaster`，对每个 Mesh 逐一检测包围球。当场景中有数百个行星 + 数千个卫星 + 5000 个小行星时，每帧鼠标移动触发的射线检测会成为瓶颈。
- **解决方案**: 引入 BVH (Bounding Volume Hierarchy) 加速射线检测
- **参考资料**:
  - [three-mesh-bvh](https://github.com/gkjohnson/three-mesh-bvh) — Three.js BVH 加速库，可将射线检测速度提升 100x
  - R3F 集成: `<Bvh>` 组件包裹场景即可全局加速
  - **注意**: InstancedMesh 的射线检测默认逐实例遍历，大量实例时必须用 BVH 或自定义空间索引

#### 5.2.6 Render Target / FBO (帧缓冲对象 / 离屏渲染)

- **重要程度**: **高** (小地图、虫洞预览、多视图必需)
- **缺口说明**: PLAN §4.9 提到小地图用独立 `OrthographicCamera`，PLAN §8.1.2 提到虫洞中心显示目标星区缩略景象。但未提及底层的 Render Target 技术——这些功能**不是**简单的多 Camera 渲染，需要将场景渲染到离屏纹理再贴到 2D HUD 或 Torus 中心。
- **参考资料**:
  - [Three.js 文档: WebGLRenderTarget](https://threejs.org/docs/#api/en/renderers/WebGLRenderTarget)
  - [R3F createPortal + useFBO](https://docs.pmnd.rs/react-three-fiber/advanced/primitives#primitives) — R3F 离屏渲染
  - [drei `useFBO` Hook](https://github.com/pmndrs/drei#usefbo) — 便捷的帧缓冲对象

#### 5.2.7 Shader Warmup / 预编译

- **重要程度**: **中高** (首帧卡顿的主要原因)
- **缺口说明**: 当用户首次接近某颗行星触发 Voronoi Shader，或首次进入黑洞触发 Raymarching Shader 时，GPU 需要即时编译 GLSL 程序，会造成 **100-500ms 的帧冻结**。PLAN 未提及任何预编译策略。
- **解决方案**:
  - 场景初始化时用不可见 Mesh "热身"所有 Shader（渲染 1 帧后隐藏）
  - 利用 `WebGLRenderer.compile(scene, camera)` 提前编译
  - KHR_parallel_shader_compile 扩展（可查询编译完成状态）
- **参考资料**:
  - [Three.js WebGLRenderer.compile()](https://threejs.org/docs/#api/en/renderers/WebGLRenderer.compile)
  - [Chrome Blog: Shader Compilation Jank](https://developer.chrome.com/blog/shader-compilation) — Chromium 团队关于 Shader 编译卡顿的分析

---

#### 5.2.8 Web Worker / OffscreenCanvas

- **重要程度**: **中高** (主线程 CPU 密集计算会阻塞渲染)
- **缺口说明**: 以下计算应移到 Worker 线程，否则会导致帧率下降：
  - 八叉树空间索引的构建和更新
  - 斐波那契球面大量轨道坐标的批量计算
  - 文章 Markdown 解析（长文 > 10000 字时耗时明显）
  - 搜索索引的前端构建
  - 星座关联度的计算
- **参考资料**:
  - [MDN: Web Workers API](https://developer.mozilla.org/en-US/docs/Web/API/Web_Workers_API)
  - [OffscreenCanvas](https://developer.mozilla.org/en-US/docs/Web/API/OffscreenCanvas) — 将 Three.js 渲染移到 Worker（实验性，兼容性有限）
  - [Comlink](https://github.com/GoogleChromeLabs/comlink) — Google 出品的 Worker 调用简化库

#### 5.2.9 React/R3F 渲染优化模式

- **重要程度**: **高** (R3F 最常见的性能陷阱)
- **缺口说明**: PLAN 中完全未提及 React 层面的优化。R3F 的每次 React re-render 都会重新遍历 3D 场景树，如果组件结构不当，简单的 Zustand 状态变更（如 HUD 面板开关）就会导致所有天体组件重新渲染。
- **核心模式**:
  ```jsx
  // 错误: 订阅整个 store，任何状态变更都触发 re-render
  const store = useCosmicStore()

  // 正确: 仅订阅需要的字段（selector）
  const focusedBody = useCosmicStore(s => s.focusedBody)

  // 关键: 3D 组件中用 useFrame 做动画，而非 useState 触发 re-render
  // 错误: 每帧 setState → 每帧 re-render → 巨慢
  const [rotation, setRotation] = useState(0)
  useFrame(() => setRotation(r => r + 0.01))

  // 正确: 直接操作 ref，零 re-render
  const meshRef = useRef()
  useFrame((_, delta) => { meshRef.current.rotation.y += speed * delta })
  ```
- **参考资料**:
  - [R3F Performance Pitfalls](https://docs.pmnd.rs/react-three-fiber/advanced/pitfalls) — **必读**，官方性能陷阱指南
  - [R3F Scaling Performance](https://docs.pmnd.rs/react-three-fiber/advanced/scaling-performance) — 大规模场景优化
  - [Zustand Selector 文档](https://docs.pmnd.rs/zustand/guides/prevent-rerenders-with-use-shallow) — 防止不必要的 re-render

#### 5.2.10 Throttle / Debounce 全局策略

- **重要程度**: **中高**
- **缺口说明**: PLAN 仅在草稿自动保存一处提到 debounce。以下场景需要系统性的节流策略：
  - `onPointerMove` — 3D 场景中鼠标移动触发射线检测（应 throttle 至 ~30fps）
  - `window.resize` — Canvas 尺寸调整（应 debounce 200ms）
  - WebSocket 位置同步 — PLAN §11.4 提到每 2 秒同步，但未提及客户端采样节流
  - 搜索输入 — 搜索栏打字触发 API（应 debounce 300ms）
  - 滚动事件 — 文章面板滚动进度计算（应 throttle 至 ~10fps）
- **参考资料**:
  - [lodash.throttle / lodash.debounce](https://lodash.com/docs/4.17.15#throttle) — 工业标准实现
  - [scheduler (React 内部)](https://github.com/nicolo-ribaudo/tc39-proposal-scheduler) — requestIdleCallback 封装

#### 5.2.11 Geometry Merge / BufferGeometry 优化

- **重要程度**: **中** (减少 Draw Call 的进阶手段)
- **缺口说明**: InstancedMesh 适用于同一几何体的多实例。但以下场景有**不同几何体**需要合并：
  - 多条轨道环线（不同半径的 Ring）合并为一个 BufferGeometry
  - 星座连线合并为单个 LineSegments
  - 星系粒子 + 星云粒子合并
- **参考资料**:
  - [Three.js BufferGeometryUtils.mergeGeometries](https://threejs.org/docs/#examples/en/utils/BufferGeometryUtils.mergeGeometries)
  - [Three.js 示例: geometry merge](https://threejs.org/examples/#webgl_geometry_merge)

#### 5.2.12 Texture Atlas (纹理图集)

- **重要程度**: **中**
- **缺口说明**: 每个行星如果使用独立纹理（即使是程序化的），每次切换纹理都是一次 GPU state change。将多个行星的程序化纹理 bake 到一张大纹理上可减少状态切换。
- **参考资料**:
  - [Three.js 示例: texture atlas](https://threejs.org/examples/#webgl_geometry_minecraft_ao)
  - **替代方案**: 项目使用程序化 Shader 生成纹理（seed 驱动），可共享同一个 ShaderMaterial 实例，仅通过 uniform/attribute 传入不同 seed，天然避免纹理切换 — **这种情况下图集不是必需的**

---

#### 5.2.13 帧预算管理 (Frame Budget)

- **重要程度**: **中**
- **缺口说明**: 60fps 要求每帧预算 16.67ms。PLAN 提到了目标帧率但未提及如何分配帧预算：
  - 物理/动画更新: ≤ 3ms
  - 射线检测: ≤ 1ms
  - React 协调: ≤ 2ms
  - GPU 渲染: ≤ 10ms
  - 余量: ~1ms
- **如果某帧超标**: 应延迟非关键工作到下一帧（如卫星重排动画分批执行）
- **参考资料**:
  - [requestIdleCallback](https://developer.mozilla.org/en-US/docs/Web/API/Window/requestIdleCallback) — 空闲时执行低优先级工作
  - [Chrome DevTools: Performance Panel](https://developer.chrome.com/docs/devtools/performance/) — 帧预算分析工具

---

#### 5.2.14 Occlusion Culling (遮挡剔除)

- **重要程度**: **低中** (深空场景遮挡较少，但黑洞/星系内部有用)
- **缺口说明**: Frustum Culling 只剔除视野外的物体。当摄像机进入某个星系内部时，其他星系被当前星系遮挡，理论上无需渲染。但深空场景多数天体是半透明的（星云、粒子），遮挡剔除的收益有限。
- **建议**: 以**手动区域管理**代替自动遮挡剔除 — 摄像机进入星系 A 时卸载远距离星系 B/C 的精细组件，仅保留远景光点
- **参考资料**:
  - [three-gpu-pathtracer 中的 BVH 遮挡](https://github.com/gkjohnson/three-gpu-pathtracer)
  - 更实用: drei `<Detailed>` 组件实现基于距离的简单剔除

---

#### 5.2.15 动画队列 / 帧分摊 (Work Distribution)

- **重要程度**: **中高**
- **缺口说明**: 以下场景会在**单帧**内产生大量计算，导致帧率骤降：
  - 搜索结果返回 50 个天体同时高亮 — 50 个 emissive 材质同时变更
  - 评论排序切换 — 100 颗卫星同时重算轨道
  - WebSocket 断线重连后批量补偿动画
  - 引力坍缩选中 20 个小行星同时开始移动
- **解决方案**: 将批量操作分摊到多帧执行
  ```js
  // 每帧只处理 N 个对象的更新
  const BATCH_SIZE = 10
  let cursor = 0
  useFrame(() => {
    const end = Math.min(cursor + BATCH_SIZE, items.length)
    for (let i = cursor; i < end; i++) {
      updateItem(items[i])
    }
    cursor = end >= items.length ? 0 : end
  })
  ```
- **参考资料**:
  - GSAP `stagger` 动画（PLAN §6.10 提到但未从帧预算角度分析）
  - [React Concurrent Mode / startTransition](https://react.dev/reference/react/startTransition) — 非紧急 UI 更新降优先级

#### 8.1.1 斐波那契球面分布 → 用错了场景

- **当前方案**: PLAN_02 §3.4、PLAN_04 §6.6 使用斐波那契球面算法为评论卫星分配轨道坐标 `(θ, φ)`
- **问题**:
  - 斐波那契球面将点**均匀分布在球面上**——但卫星不应该分布在球面上，卫星应该沿**轨道环带**运行（就像土星环、木星卫星系统）
  - 均匀球面分布意味着卫星会出现在行星的正上方、正下方、任何角度——视觉上像一团蜜蜂围着球飞，而不是优雅的轨道系统
  - 真实效果：50 颗卫星均匀铺在球面上 = 稀疏的噪点球，毫无"行星环"的壮观感
- **改造方案**: **分层轨道环带分配算法**（需自研）
  ```
  算法: LayeredOrbitalAllocation(n_satellites)

  1. 计算环带数 = ceil(n_satellites / SATELLITES_PER_RING)  // 每环 8-12 颗
  2. 为每个环带分配:
     - orbit_radius: 从洛希极限内边缘开始，每层递增 Δr（带随机抖动 ±10%）
     - orbit_inclination: 第一环 0°（赤道面），后续环依次倾斜 ±5°~15°（交错排列）
  3. 环内卫星均匀分配相位角: phase = 2π * i / n_in_ring + 随机抖动
  4. 当 n > 50 时:
     - 环间距收窄 → 视觉上形成连续的"行星环"
     - 最外层切换为扁平粒子带（InstancedMesh Points 替代球体 Mesh）
  ```
- **效果**: 少量评论 = 几条清晰的轨道环；大量评论 = 壮观的土星环结构

#### 8.1.2 简单三角函数圆周轨道 → 物理上不真实

- **当前方案**: PLAN_03 §4.4、PLAN_04 §6.1 使用 `x = r * cos(time * speed + offset)` 计算公转
- **问题**:
  - 所有轨道都是**正圆**，匀速运动——看起来像钟表齿轮，缺乏太空感
  - 违反开普勒第二定律（面积速度定律）：真实轨道近日点快、远日点慢
  - 所有行星以固定速度转圈，大行星和小行星速度相同——不符合直觉
- **改造方案**: **简化开普勒椭圆轨道**
  ```
  改造要点:
  1. 轨道形状: 圆 → 椭圆 (引入离心率 e = 0.05~0.3)
     x = a * cos(E) - c
     z = b * sin(E)
     其中 a = 半长轴, b = a*sqrt(1-e²), c = a*e

  2. 速度变化: 匀速 → 面积速度恒定 (开普勒方程)
     M = E - e * sin(E)    // M = mean anomaly = time * speed
     通过 Newton-Raphson 迭代 3 次求解 E (eccentric anomaly)

  3. 质量-速度关联: 大行星（长文章）轨道半径大、速度慢
     speed = K / sqrt(orbit_radius³)  // 开普勒第三定律简化
  ```
  - Newton-Raphson 3 次迭代 CPU 开销极低（< 0.001ms/天体），不影响性能
- **效果**: 行星轨道有了椭圆感和速度变化，近距离经过时会微微加速掠过，远处则缓慢飘行

#### 8.1.3 编辑距离计算文章相似度 → 完全不适用

- **当前方案**: PLAN_03 §4.8 使用 Levenshtein 编辑距离计算文章标题相似度，用于星座关联
- **问题**:
  - 编辑距离是**字符级**度量，衡量两个字符串之间的字符变换次数
  - 标题"Go 并发编程实战"和"Golang goroutine 与 channel"编辑距离很大，但语义极其相关
  - 标题"我的第一篇"和"我的第二篇"编辑距离很小，但内容可能完全无关
  - 这个算法用在了**完全错误的场景**——文章相似度需要语义级别的比较，不是字符级
- **替代方案（按优先级）**:
  1. **pgvector 余弦相似度**（最佳，见 §7.3.1）——语义级相关性
  2. **Jaccard 标签交集系数**（简单有效）——`|tags_A ∩ tags_B| / |tags_A ∪ tags_B|`
  3. **TF-IDF + 余弦相似度**（传统但有效）——基于词频的文档相似度
  - 完全**删除**编辑距离，它在这个场景下是误用

#### 8.1.4 AES-256-GCM 直接使用用户输入作为密钥 → 安全缺陷

- **当前方案**: PLAN_05 §10.3 中"密钥由用户输入"直接用于 AES-256 解密
- **问题**:
  - AES-256 需要 256 位（32 字节）密钥。用户输入的是密码/口令（"mypassword123"），通常只有 8-20 个 ASCII 字符
  - 直接将短密码填充/截断为 32 字节 → **密钥空间极小**，暴力破解毫秒级完成
  - 这是一个**严重安全漏洞**——加密形同虚设
- **改造方案**: **PBKDF2 / Argon2id 密钥派生**
  ```
  正确流程:
  1. 用户输入口令 passphrase
  2. 从存储中读取该文章的 salt（随机 16 字节，明文存储，每篇文章不同）
  3. 派生密钥: key = Argon2id(passphrase, salt, time=3, memory=64MB)
     或: key = PBKDF2(passphrase, salt, iterations=600000, hash=SHA256)
  4. 用派生的 key 进行 AES-256-GCM 解密

  前端（Web Crypto API）:
  const keyMaterial = await crypto.subtle.importKey("raw", encode(passphrase), "PBKDF2", false, ["deriveKey"])
  const key = await crypto.subtle.deriveKey(
    { name: "PBKDF2", salt, iterations: 600000, hash: "SHA-256" },
    keyMaterial,
    { name: "AES-GCM", length: 256 },
    false,
    ["decrypt"]
  )
  ```
- **影响**: 必须修改 PLAN_05 §10.3 和 §10.4 的加解密流程，增加 salt 存储字段

#### 8.1.5 UUID 作为所有主键 → 索引性能差

- **当前方案**: PLAN_02 §3.1 所有表使用 `UUID PRIMARY KEY`
- **问题**:
  - UUID v4 完全随机 → B-Tree 索引频繁页分裂，写入性能随数据量增长退化
  - UUID v4 不含时间信息 → 无法通过 ID 推断创建顺序，排序必须依赖额外的 `created_at` 索引
  - 128 位 (36 字符字符串) 占用空间是 64 位整数的 4 倍，所有外键引用同样膨胀
- **改造方案**: **ULID 或 UUIDv7**
  ```
  ULID (Universally Unique Lexicographically Sortable Identifier):
  - 128 位，与 UUID 兼容（可存入 PostgreSQL UUID 类型）
  - 前 48 位 = 毫秒级时间戳 → 按 ID 排序 = 按时间排序
  - 后 80 位 = 随机数 → 同一毫秒内仍唯一
  - B-Tree 友好: 新 ID 总是追加到索引末尾（近似自增），零页分裂

  UUIDv7 (RFC 9562, 2024):
  - 与 ULID 原理相同，是 UUID 标准的正式版本
  - PostgreSQL 15+ 原生支持 gen_random_uuid() 但仅 v4
  - Go: github.com/google/uuid 已支持 uuid.NewV7()
  ```
  - `ORDER BY id` 即 `ORDER BY created_at`，可**删除** `created_at` 上的部分索引
- **影响**: 所有 `id UUID` 改为使用 UUIDv7 生成策略，不需要改列类型

#### 8.1.6 LOD 硬切换 → 视觉跳变

- **当前方案**: PLAN_03 §4.1、PLAN_04 §6.2 的 LOD 在距离阈值处直接切换模型
- **问题**:
  - Three.js 原生 `LOD` 对象在切换距离处瞬间替换模型——用户会看到天体"闪烁"一下变精细或变粗糙
  - 当摄像机在切换距离附近来回移动时，LOD 反复切换，更加刺眼
- **改造方案**: **交叉淡入淡出 LOD（Dithered LOD / Cross-fade LOD）**
  ```
  思路:
  1. 在切换距离的 ±10% 范围内设置过渡带
  2. 过渡带内同时渲染两个 LOD 级别
  3. 高精度 LOD: opacity 从 1 → 0 (Shader 中 dither 透明)
  4. 低精度 LOD: opacity 从 0 → 1
  5. 过渡完成后移除旧 LOD

  优化: 使用 Shader 中的 dithered transparency（棋盘格透明）
  代替真透明——无需开启 alpha blend，无排序问题:
    float dither = fract(sin(dot(gl_FragCoord.xy, vec2(12.9898, 78.233))) * 43758.5453);
    if (dither > fadeProgress) discard;
  ```
- **参考**: Unreal Engine 的 Dithered LOD Transition 即使用此技术

#### 8.1.7 蜜罐 + 提交时间 反垃圾 → 已被现代爬虫轻松绕过

- **当前方案**: PLAN_04 §6.12 使用蜜罐字段 + 提交时间 < 3秒检测
- **问题**:
  - 现代 headless browser 爬虫（Playwright/Puppeteer）**不会**填写隐藏字段——蜜罐失效
  - GPT 驱动的垃圾评论机器人会模拟人类等待时间——时间检测失效
  - 关键词黑名单维护成本高且无法覆盖变体（"v1@gra" 绕过 "viagra"）
- **改造方案**: **多层防御 + Proof of Work**
  ```
  层级防御:

  1. 前端 Proof of Work (PoW):
     - 提交评论前要求浏览器解一个小型 SHA-256 哈希挑战
     - 如: 找到 nonce 使得 SHA256(comment_id + nonce) 前 18 位为 0
     - 正常设备 ~200ms 完成，对用户无感
     - 大规模垃圾发送者成本 × 1000（每条都需要 200ms 计算）
     - 参考: altcha.org (开源 PoW 验证码)

  2. 浏览器指纹行为分析:
     - 记录用户从打开页面到提交的: 鼠标轨迹熵、键盘输入节奏、滚动行为
     - 机器人的行为熵极低（完美直线鼠标、均匀键入间隔）
     - 无需传到后端——前端计算一个 behavior_score，低于阈值拒绝提交

  3. 后端频率 + 内容分析:
     - 同 IP 滑动窗口限流（保留）
     - 评论内容与文章相关度检测（TF-IDF 交集 > 阈值才通过）
     - 重复内容检测（SimHash 指纹去重）
  ```

#### 8.1.8 HyperLogLog 统计阅读量 → 个人博客场景下杀鸡用牛刀

- **当前方案**: PLAN_02 §2.13 使用 Redis HyperLogLog 统计文章阅读量
- **问题**:
  - HyperLogLog 设计用于**亿级基数估计**（误差 0.81%），1 亿独立访客的内存占用仅 12KB
  - 个人博客的热门文章阅读量通常在 1K-100K 量级
  - 在这个量级下 HyperLogLog 的误差（~0.81%）= ±810 次阅读，可能比真实值偏差更大
  - 而且 HyperLogLog **不支持查询某个特定访客是否已计数**——无法实现"同 IP 24 小时不重复"
- **替代方案**:
  ```
  方案 A: Redis SET (精确去重，阅读量 < 10 万时)
  - SADD view:{slug}:{date} {ip_hash}
  - SCARD view:{slug}:{date} → 精确阅读数
  - TTL 24h 自动过期
  - 内存: 10万 IP hash × 8 bytes ≈ 800KB/篇/天

  方案 B: Bloom Filter (近似去重，阅读量 > 10 万时升级)
  - BF.ADD view:{slug}:{date} {ip_hash}
  - BF.EXISTS → 判断是否已计数
  - 误判率 1%，内存仅 SET 的 1/8
  - Redis Stack 原生支持 BF.* 命令

  策略: 默认用 SET；当某篇文章的 SET 大小超过阈值时自动迁移到 Bloom Filter
  ```

  ```
     - 数据/图表密集 → 几何/晶体纹理

  4. 情感分析（可选，LLM 辅助）:
     - 积极/激昂 → 高亮度、强自发光
     - 沉思/深度 → 低饱和度、冷光晕
     - 紧张/悬疑 → 暗红色大气、表面裂纹纹理

  5. 活跃度动态参数:
     - 近期有评论 → 大气层活跃（云层流动加速）
     - 长期无人访问 → 表面风化纹理加深（与虫洞遗迹呼应）
  ```
- **为什么需要发明**: 现有的程序化生成都是基于纯随机 seed 的无语义生成。将内容语义映射到视觉美学是一个**跨域映射问题**，没有通用解

#### 8.2.2 [发明] 自适应轨道再平衡算法 (Adaptive Orbit Rebalancing)

- **问题**: 评论增删后，卫星轨道需要动态调整。当前方案（斐波那契球面重新分配）会导致**所有**卫星位置突变
- **现状**: 没有现成算法能在保持大多数卫星位置稳定的前提下，为新卫星找到最优位置
- **需要发明**:
  ```
  AdaptiveOrbitRebalancer:

  新增卫星:
  1. 扫描现有轨道环带，找到"最宽空隙"（相邻卫星间角度差最大的位置）
  2. 如果最宽空隙 > 最小安全间距 → 直接插入空隙中点，其他卫星不动
  3. 如果所有环带都满 → 新开一层环带（半径 = 最外层 + Δr）
  4. 如果总层数超过阈值 → 触发"行星环模式"，所有卫星压缩到窄带

  删除卫星:
  1. 仅移除该卫星，形成空隙
  2. 空隙两侧的卫星向中心各 Lerp 微调 5%（填补空隙但不大幅移动）
  3. 如果整层仅剩 1-2 颗 → 将它们迁移到相邻层（合并稀疏层）

  批量变更（排序切换）:
  1. 计算目标位置集合
  2. 使用匈牙利算法（最优指派）将现有卫星指派到最近的目标位置
     → 最小化全局移动距离，大多数卫星只需微调
  3. GSAP stagger 动画执行迁移

  关键约束:
  - 单次操作最多移动 20% 的卫星（稳定性保障）
  - 移动距离超过半圈的卫星走弧线而非直线（防止穿越行星）
  ```
- **为什么需要发明**: 这是一个带约束的在线动态布局问题（online constrained layout），既要视觉美观又要最小扰动，没有现成库

#### 8.2.3 [发明] 注意力加权 LOD (Attention-Weighted LOD)

- **问题**: 标准 LOD 仅基于摄像机距离决定细节级别。但在这个项目中，"重要性"不只取决于距离
- **现状**: Three.js 和所有 3D 引擎的 LOD 都只看距离
- **需要发明**:
  ```
  AttentionWeightedLOD:

  计算每个天体的注意力权重 W:

  W = w_dist × DistanceFactor      // 基础距离因子（现有 LOD 逻辑）
    + w_gaze × GazeFactor           // 视线方向因子: 摄像机 forward 与天体方向的点积
    + w_pop  × PopularityFactor     // 人气因子: 评论数/阅读量归一化
    + w_rec  × RecencyFactor        // 新鲜度因子: 发布时间越近权重越高
    + w_int  × InteractionFactor    // 交互因子: 用户本次会话悬停/点击过的天体提权
    + w_nav  × NavigationFactor     // 导航因子: 当前路由指向的天体全权重

  LOD 级别 = quantize(W, [threshold_high, threshold_mid, threshold_low])

  使用场景:
  - 用户看向一颗远处的热门行星 → 即使距离远，因为人气高 + 视线方向对准，
    仍然渲染中等 LOD（而非标准 LOD 的最低级光点）
  - 用户刚刚悬停过的行星 → 即使摄像机已移开，5 秒内保持中等 LOD（避免刚看过就降级）
  - 摄像机正在飞向目标 → 目标天体提前加载高 LOD（而非到达阈值才切换）
  ```
- **为什么需要发明**: 传统 LOD 是纯几何判断。在"每个天体都承载内容"的博客场景中，LOD 需要理解内容重要性，这是传统游戏引擎不需要解决的问题

#### 8.2.4 [发明] 渐进式场景揭示算法 (Progressive Scene Revelation)

- **问题**: 用户首次打开页面时，全宇宙数百个天体同时初始化和渲染 → 大量资源竞争，首屏体验差
- **现状**: 现有加载策略只区分"加载中"和"加载完成"两个状态，缺乏优先级调度
- **需要发明**:
  ```
  ProgressiveSceneRevelation:

  第一批 (0-500ms): 即时渲染
  - 深空背景（Skybox / 星点粒子）→ 最低成本，立即呈现沉浸感
  - 摄像机初始位置可见的 2-3 个最近星系（仅光团级 LOD）

  第二批 (500ms-1.5s): 首屏关键
  - 初始视野内的所有星系升级到粒子级 LOD
  - 脉冲星（宇宙中心）开始闪烁
  - 如果用户直接访问 /post/slug → 跳过宇宙全景，只加载目标行星

  第三批 (1.5s-3s): 交互就绪
  - 摄像机附近星系内的行星开始渲染（程序化纹理编译）
  - 射线检测系统激活（此前鼠标事件不响应，避免未加载天体的空点击）

  第四批 (3s-10s): 后台填充
  - 视野外的星系逐个加载（每帧加载 1 个，不阻塞渲染）
  - 虫洞、黑洞等特殊天体延迟加载
  - 所有 Shader 预热（§1.20）

  第五批 (10s+): 完全就绪
  - 小行星带粒子填充
  - 星座连线计算和渲染
  - 非关键后处理效果启用（God Rays, SSAO 等）

  调度器:
  - 使用 requestIdleCallback 在浏览器空闲时执行低优先级加载
  - 用户开始交互（鼠标移动/滚轮）→ 立即暂停后台加载，优先响应交互
  - 根据设备性能分档调整各批次时间（低端设备每批延长 50%）
  ```
- **为什么需要发明**: 传统的 loading → loaded 二元模型不适合 3D 重型应用。需要一个内容感知的分优先级调度器，理解"用户此刻最需要看到什么"

#### 8.2.5 [发明] 宇宙布局稳定性维护算法 (Cosmic Layout Stability)

- **问题**: 博客持续运营，文章不断增加。如果使用力导向图或 UMAP 布局，每次新增文章都会导致全局坐标重新计算 → 老文章的行星位置漂移 → 老访客回来发现"宇宙变了"，丧失空间记忆
- **现状**: 力导向图和 UMAP 都是全局优化算法，无法保证增量稳定性
- **需要发明**:
  ```
  CosmicLayoutStabilizer:

  原则: "旧天体位置神圣不可侵犯，新天体适应旧布局"

  1. 已发布文章的坐标一经确定，写入 DB 后永不自动变更
     - 管理后台可手动拖拽调整（显式操作）

  2. 新文章定位算法:
     a. 计算新文章与所有现有文章的语义相似度（pgvector）
     b. 找到 Top-3 最相似文章的坐标
     c. 新文章坐标 = Top-3 坐标的加权质心 + 排斥偏移
        position = weighted_centroid(top3) + repulsion_offset
     d. 排斥偏移: 确保与所有现有天体保持最小距离 d_min
        用简单的斥力迭代（10 次）推开重叠:
        for each existing body within d_min:
          offset += normalize(new_pos - body.pos) * (d_min - distance) * 0.5

  3. 星系边界弹性扩展:
     - 新文章的目标坐标超出所属星系当前边界 → 星系边界平滑扩展（半径 +10%）
     - 其他星系不受影响

  4. 稳定性保证:
     - 新增 1 篇文章 → 0 篇旧文章移动
     - 删除 1 篇文章 → 仅该位置空出，不做填补（宇宙中有"空地"是自然的）
     - 星系合并/拆分 → 仅管理员手动操作时触发全局重布局
  ```
- **为什么需要发明**: 这是一个**增量式约束布局**问题——在保持历史布局不变的前提下为新元素找到最优位置。现有的布局算法都是批量全局优化的

#### 8.2.6 [发明] 摄像机意图预测器 (Camera Intent Predictor)

- **问题**: 当前所有摄像机动画都是**响应式**的——用户点击后才开始加载目标天体的资源（高清纹理、评论数据等）。导致到达后需要等待加载
- **需要发明**:
  ```
  CameraIntentPredictor:

  通过用户行为推断"下一步最可能去哪"，提前预加载:

  信号权重:
  1. 鼠标轨迹方向 (0.3): 鼠标持续向左移动 → 预加载左侧天体数据
  2. 视线停留 (0.4): 摄像机对准某方向 > 2 秒 → 该方向天体大概率是下一目标
  3. 导航上下文 (0.2): 用户正在浏览"Go"星系的文章 → 预加载同星系下一篇
  4. 热力统计 (0.1): 大多数访客从 A 文章跳转到 B 文章 → 预加载 B

  预加载策略:
  - 当预测置信度 > 0.6 时:
    ① 开始请求目标天体的 GraphQL 详情数据（评论列表等）
    ② 目标天体的 LOD 提前升一级（注意力加权 LOD 的 NavigationFactor 生效）
    ③ 目标天体的 Shader Material 提前编译（如尚未编译）
  - 预测错误的代价: 仅浪费一次轻量 API 请求 + 少量 GPU 内存
  - 预测正确的收益: 用户到达时零等待，数据和视觉都已就绪
  ```
- **为什么需要发明**: 传统 Web 的 prefetch（`<link rel="prefetch">`）只对页面导航有效。3D 空间中的资源预加载需要理解空间意图，这是全新的问题域

### 7.1 图形学前沿

#### 7.1.1 Curl Noise 流体粒子场 — 星云/星际尘埃

- **适用场景**: 星系星云（PLAN_03 §4.1）、小行星带漂移（PLAN_04 §7.1）、虫洞跃迁粒子（PLAN_05 §8.2）
- **当前方案**: 粒子使用简单的随机位移 + 三角函数漂浮，运动轨迹缺乏流体感
- **先进替代**: **Curl Noise**（旋度噪声）——对 3D Simplex Noise 场求旋度（curl），生成**无散度**的速度场。粒子沿该速度场运动时，呈现类似流体、烟雾的有机涡旋效果，且**永不穿模**（无散度 = 不压缩 = 粒子不会聚集在一点）
- **视觉效果**: 星云粒子会呈现如同真实星际气体的卷曲、涡旋运动，而非无生命的随机抖动
- **核心 GLSL**:
  ```glsl
  vec3 curlNoise(vec3 p) {
    float e = 0.1;
    vec3 dx = vec3(e, 0, 0), dy = vec3(0, e, 0), dz = vec3(0, 0, e);
    float x = snoise(p + dy).z - snoise(p - dy).z - snoise(p + dz).y + snoise(p - dz).y;
    float y = snoise(p + dz).x - snoise(p - dz).x - snoise(p + dx).z + snoise(p - dx).z;
    float z = snoise(p + dx).y - snoise(p - dx).y - snoise(p + dy).x + snoise(p - dy).x;
    return normalize(vec3(x, y, z) / (2.0 * e));
  }
  ```
- **参考资料**:
  - Bridson, R. et al. "Curl-Noise for Procedural Fluid Flow" (SIGGRAPH 2007) — [原始论文](https://www.cs.ubc.ca/~rbridson/docs/bridson-siggraph2007-curlnoise.pdf)
  - [Shadertoy: Curl Noise Particles](https://www.shadertoy.com/view/4dBBWR)
  - [Three.js Curl Noise 示例 (Codrops)](https://tympanus.net/codrops/2024/05/29/creating-curl-noise-particle-effects-with-three-js/)
- **实施建议**: 替换 `Starfield.tsx` 和 `Galaxy.tsx` 的粒子运动逻辑，在 Vertex Shader 中通过 Curl Noise 计算每帧位移。GPU 端执行，零 CPU 开销

#### 7.1.2 Volumetric Light Scattering (体积光散射 / God Rays)

- **适用场景**: 脉冲星射线束（PLAN_05 §9.1）、黑洞吸积盘光晕、虫洞中心光芒
- **当前方案**: 使用 `UnrealBloom` 后处理做泛光，但 Bloom 是全屏均匀的，无法模拟光线穿过介质的定向散射
- **先进替代**: **屏幕空间体积光散射**（Screen-space Volumetric Light Scattering）——从光源位置向屏幕空间做径向采样，叠加光线在空间中的散射衰减，产生"上帝光线"效果
- **视觉效果**: 脉冲星的两极射线不再只是发光的圆锥体，而是能看到光线穿透星际尘埃时的散射条纹，极具真实感和震撼力
- **参考资料**:
  - [GPU Gems 3 - Chapter 13: Volumetric Light Scattering](https://developer.nvidia.com/gpugems/gpugems3/part-ii-light-and-shadows/chapter-13-volumetric-light-scattering-post-process) — NVIDIA 经典教程
  - [Shadertoy: God Rays](https://www.shadertoy.com/view/XdfXRj) — 可运行的 GLSL 实现
  - [Three.js 示例: postprocessing godrays](https://threejs.org/examples/#webgl_postprocessing_godrays) — 官方示例
  - [pmndrs/postprocessing GodRaysEffect](https://github.com/pmndrs/postprocessing) — R3F 生态可直接使用
- **实施建议**: 仅对脉冲星和活跃黑洞启用；通过自适应画质系统在低端设备关闭

#### 7.1.3 Reaction-Diffusion (反应扩散 / 图灵斑图)

- **适用场景**: 行星表面纹理程序化生成（PLAN_03 §4.2）、降维微观视界的"细胞"纹理（PLAN_03 §5.3）
- **当前方案**: 使用 Voronoi + Simplex Noise 生成地貌纹理，效果优秀但相对常规
- **先进替代**: **Gray-Scott 反应扩散模型**——模拟两种化学物质在空间中的扩散与反应，自发形成**图灵斑图**（斑点、条纹、迷宫、珊瑚状结构）。这些图案是自然界中生物体表面纹理的数学本源（斑马条纹、豹纹、珊瑚结构）
- **视觉效果**: 行星表面呈现出活生生的、不断演化的有机纹理。用在降维微观视界中，文章文字镶嵌在如同活体细胞壁一样蠕动的纹理上——比静态 Voronoi 震撼得多
- **核心 GLSL**:
  ```glsl
  // Gray-Scott Model: dU/dt = Du∇²U - UV² + f(1-U),  dV/dt = Dv∇²V + UV² - (f+k)V
  // 在 Fragment Shader 中用 ping-pong FBO 迭代
  float laplacianU = textureOffset(uState, uv, ivec2(1,0)).r + textureOffset(uState, uv, ivec2(-1,0)).r
                   + textureOffset(uState, uv, ivec2(0,1)).r + textureOffset(uState, uv, ivec2(0,-1)).r
                   - 4.0 * U;
  float newU = U + (Du * laplacianU - U * V * V + feed * (1.0 - U)) * dt;
  float newV = V + (Dv * laplacianV + U * V * V - (feed + kill) * V) * dt;
  ```
- **参考资料**:
  - [Karl Sims - Reaction Diffusion Tutorial](https://karlsims.com/rd.html) — 最佳入门解释 + 交互式 Demo
  - [pmndrs/react-three-fiber 中的 FBO ping-pong 技术](https://docs.pmnd.rs/react-three-fiber/advanced/primitives) — 实现 GPU 迭代的基础
  - [Shadertoy: Gray-Scott Reaction Diffusion](https://www.shadertoy.com/view/XsG3z1)
  - Turing, A.M. "The Chemical Basis of Morphogenesis" (1952) — 图灵的原始论文
- **实施建议**: 使用 FBO ping-pong 在 GPU 上迭代，CPU 零开销。预计算好一帧稳态纹理 bake 为 Texture 贴到行星上（避免实时迭代的 GPU 压力）；降维微观视界中可实时低分辨率迭代

#### 7.1.4 SDF Text Rendering (有符号距离场文字渲染)

- **适用场景**: 所有 3D 场景中的文字标签——星系名称、行星标题、虫洞年份、卫星评论者 ID
- **当前方案**: 使用 R3F `<Html>` 组件注入 DOM 文字，或使用 `<Text>` 组件（基于 Troika）
- **先进替代**: **SDF 文字渲染**——将字体字形预处理为有符号距离场纹理（每个像素存储到最近边缘的距离），渲染时在 Fragment Shader 中通过 `smoothstep` 在距离场上采样。文字在**任意缩放比例下**都锐利清晰，无锯齿、无模糊
- **视觉效果**: 从宇宙全景拉远到极近距离查看天体名称，文字始终保持完美清晰度。支持描边、发光、阴影等特效直接在 Shader 中实现，无需额外 Draw Call
- **参考资料**:
  - Valve - "Improved Alpha-Tested Magnification for Vector Textures and Special Effects" (SIGGRAPH 2007) — SDF 文字渲染的奠基论文
  - [msdf-atlas-gen](https://github.com/Chlumsky/msdf-atlas-gen) — 多通道 SDF 字体图集生成工具（MSDF 比单通道 SDF 更精确）
  - [troika-three-text](https://github.com/protectwise/troika/tree/main/packages/troika-three-text) — Three.js SDF 文字渲染库（drei `<Text>` 底层即使用此库）
  - [three-mesh-ui](https://github.com/felixmariotto/three-mesh-ui) — 3D 空间中的 SDF 文字 + UI 布局
- **实施建议**: drei `<Text>` 已经使用 Troika（基于 SDF），确认项目中一致使用 `<Text>` 而非 `<Html>` 渲染场景内文字标签即可获得 SDF 优势。对中文需要预生成字符子集的 SDF 图集

#### 7.1.5 Temporal Anti-Aliasing (TAA 时域抗锯齿)

- **适用场景**: 全局渲染质量提升——行星轮廓、轨道线、星云粒子边缘
- **当前方案**: WebGL 标准 MSAA（Multi-Sample AA），在后处理管线启用后 MSAA 失效（R3F 已知问题）
- **先进替代**: **TAA**——利用前一帧的渲染结果，通过子像素抖动 + 历史帧混合来消除锯齿。即使在后处理链之后依然有效，且对 Shader 边缘（Bloom 光晕、Voronoi 细胞壁）的抗锯齿效果远优于 MSAA
- **参考资料**:
  - [pmndrs/postprocessing SMAA](https://github.com/pmndrs/postprocessing) — 后处理兼容的抗锯齿（SMAA 比 TAA 简单，可作为替代）
  - [Playdead TAA 技术分享 (GDC 2016)](https://www.gdcvault.com/play/1022970/Temporal-Reprojection-Anti-Aliasing-in) — AAA 级 TAA 实现讲解
  - Three.js 中通过 `jitterProjectionMatrix` + 自定义后处理 Pass 实现
- **实施建议**: 画质"高"时启用 SMAA 后处理 Pass，中/低画质关闭。TAA 实现复杂，推荐用 SMAA 替代

#### 7.1.6 Screen Space Ambient Occlusion (SSAO 屏幕空间环境光遮蔽)

- **适用场景**: 行星表面接近大气突入视角时的深度感、小行星带碎石的立体感
- **当前方案**: 无环境光遮蔽，物体交界处缺乏阴影过渡，看起来"浮"在空间中
- **先进替代**: **SSAO / HBAO+**——在屏幕空间采样深度缓冲，估算每个像素被周围几何体遮挡的程度，在凹角处添加微弱阴影。让物体交界处产生自然的接触阴影，极大增强 3D 立体感
- **参考资料**:
  - [Learn OpenGL - SSAO](https://learnopengl.com/Advanced-Lighting/SSAO) — 原理和完整实现
  - [pmndrs/postprocessing SSAOEffect](https://github.com/pmndrs/postprocessing) — R3F 生态可直接使用
  - [N8AO](https://github.com/N8python/n8ao) — 高质量、高性能的 Three.js SSAO 实现
- **实施建议**: 仅在行星近距离视角（大气突入阶段）启用；远景关闭（深空场景收益低）。建议使用 N8AO 库，性能优于 pmndrs 版本

#### 7.1.7 Depth of Field (景深/散景)

- **适用场景**: 聚焦某颗行星时，远景和近景天体模糊化，营造电影级镜头感
- **当前方案**: 无景深效果，所有距离的物体同等清晰，缺乏纵深层次
- **先进替代**: **Bokeh DoF**——模拟物理镜头的焦点范围，焦点外的物体产生圆形光斑（散景），焦点处保持锐利。从宇宙全景聚焦到某颗行星时，背景星系自然模糊为光斑
- **参考资料**:
  - [pmndrs/postprocessing DepthOfFieldEffect](https://github.com/pmndrs/postprocessing) — R3F 可用
  - [Three.js 示例: postprocessing dof](https://threejs.org/examples/#webgl_postprocessing_dof2)
- **实施建议**: 仅在摄像机飞行过渡期间动态启用（GSAP 控制焦距参数），着陆后关闭。画质"低"时禁用

### 7.2 物理模拟前沿

#### 7.2.1 N-Body Gravitational Simulation (N 体引力模拟)

- **适用场景**: 小行星带漂移（PLAN_04 §7.1）、卫星群运动（PLAN_04 §6.1）、引力坍缩动效（PLAN_04 §7.3）
- **当前方案**: 卫星和小行星使用预设参数方程公转（`cos/sin`），运动完全确定性，缺乏天体间的引力交互
- **先进替代**: **Barnes-Hut N 体模拟**——将空间用八叉树划分，远距离天体群近似为单个质心点，将 O(n²) 引力计算降至 O(n log n)。天体之间产生真实的引力扰动：卫星靠近时互相排斥、经过大行星时轨道弯曲、小行星被引力捕获
- **视觉效果**: 卫星群不再是机械的圆周运动，而是有微妙摄动的"活"轨道。引力坍缩不再是预设动画，而是真正的物理碰撞聚合
- **参考资料**:
  - Barnes, J. & Hut, P. "A hierarchical O(N log N) force-calculation algorithm" (Nature, 1986) — 原始论文
  - [WebGL N-body 模拟](https://github.com/nickslevine/nbmr) — GPU 加速的 N-body 浏览器实现
  - [Shadertoy: Galaxy N-body](https://www.shadertoy.com/view/4dXBRH) — GPU Compute 实现
- **实施建议**: 不建议全量替换现有方程。建议仅在引力坍缩动效和卫星近距离交互时启用**局部 N-body**（参与天体 < 100），其余仍用参数方程。GPU Compute Shader 实现（需 WebGPU，见 §7.4）

#### 7.2.2 Boids 群集算法 (Craig Reynolds)

- **适用场景**: 在线用户飞船编队（PLAN_06 §11.4）、小行星带"鱼群"式漂移
- **当前方案**: 在线用户飞船互相独立，多人聚集时只是重叠在一起；小行星随机漂移无群体行为
- **先进替代**: **Boids 算法**——每个个体遵循三条简单规则：① 分离（避免碰撞）② 对齐（方向趋同）③ 凝聚（向群体中心靠拢）。涌现出高度有机的群集行为
- **视觉效果**: 多名在线用户的飞船会自动形成编队阵型，像太空舰队一样整齐又不死板。小行星带的碎石以鱼群般的节奏集体漂移
- **参考资料**:
  - Reynolds, C. "Flocks, Herds and Schools: A Distributed Behavioral Model" (SIGGRAPH 1987) — [经典论文](https://www.red3d.com/cwr/boids/)
  - [Three.js Boids 示例](https://threejs.org/examples/#webgl_gpgpu_birds) — GPU 加速群集（官方示例）
  - [WebGPU Boids Compute Shader](https://webgpu.github.io/webgpu-samples/?sample=computeBoids) — WebGPU 官方示例
- **实施建议**: 在线用户飞船数量少（< 30），可 CPU 实现。小行星带如启用群集行为需 GPU Compute（见 §7.4）

#### 7.2.3 Verlet Integration (韦尔莱积分) — 弹性连接与约束

- **适用场景**: 星座关联连线（PLAN_03 §4.8）的弹性拉伸、评论线程子卫星链（PLAN_04 §6.7）
- **当前方案**: 连线是刚性的直线段，卫星链是刚性的嵌套公转
- **先进替代**: **Verlet 积分**——基于位置的物理模拟（PBD），通过迭代约束投影模拟弹性绳索、弹簧连接。子卫星链用 Verlet 弹簧连接后，会像太空中的锚定绳索一样晃动、弹跳
- **参考资料**:
  - Jakobsen, T. "Advanced Character Physics" (GDC 2001) — [经典 Verlet 教程](https://www.cs.cmu.edu/afs/cs/academic/class/15462-s13/www/lec_slides/Jakobsen.pdf)
  - [Verlet.js](https://github.com/subprotocol/verlet-js) — 2D Verlet 物理库（可改 3D）
- **实施建议**: 仅用于星座连线的弹性效果和子卫星链，不用于主要天体运动

### 7.3 AI / 数据智能前沿

#### 7.3.1 pgvector 向量语义搜索

- **适用场景**: 文章相关推荐（PLAN_03 §4.8 星座关联）、全文搜索（PLAN_02 §2.9）
- **当前方案**: 文章关联度通过共同标签 + 编辑距离计算，搜索基于关键词全文检索
- **先进替代**: **向量语义搜索**——使用预训练语言模型（如 `bge-small-zh`）将每篇文章编码为 768 维向量嵌入（Embedding），存储在 PostgreSQL 的 [pgvector](https://github.com/pgvector/pgvector) 扩展中。搜索和推荐基于向量余弦相似度，能理解语义（如搜索"并发"能匹配到讨论"goroutine"的文章）
- **实现方式**:
  ```sql
  -- pgvector 扩展
  CREATE EXTENSION vector;
  ALTER TABLE celestial_bodies ADD COLUMN embedding vector(768);

  -- 语义搜索：找最相似的 5 篇文章
  SELECT title, 1 - (embedding <=> query_embedding) AS similarity
  FROM celestial_bodies
  WHERE type = 'PLANET'
  ORDER BY embedding <=> query_embedding
  LIMIT 5;
  ```
- **参考资料**:
  - [pgvector](https://github.com/pgvector/pgvector) — PostgreSQL 向量扩展
  - [BAAI/bge-small-zh](https://huggingface.co/BAAI/bge-small-zh-v1.5) — 中文嵌入模型（~100MB）
  - [Ollama](https://ollama.com/) — 本地运行嵌入模型（不依赖外部 API）
- **实施建议**: 文章发布时异步生成 Embedding（Go 调用本地 Ollama 或远程 API），存入 pgvector。星座关联直接查向量相似度，替代手动标签计算。搜索同时支持关键词 + 语义两种模式

#### 7.3.2 Force-Directed Graph Layout (力导向图布局)

- **适用场景**: 星系内行星分布（PLAN_03 §4.0）、星座关联网络可视化（PLAN_03 §4.8）
- **当前方案**: 行星位置由后端 JSONB 坐标决定（手动或随机），星座连线在固定位置的行星间绘制
- **先进替代**: **力导向图**——将天体视为图节点，关联关系为边。节点间施加斥力（库仑力）和边施加引力（弹簧力），迭代求解平衡位置。相关文章自然聚集，无关文章自动分离
- **视觉效果**: 星系内的行星分布呈现出自然的聚类结构——同标签的文章聚在一起形成小星团，不同主题的星团间保持距离。比随机分布有意义得多
- **参考资料**:
  - [d3-force-3d](https://github.com/vasturiano/d3-force-3d) — D3 力导向图的 3D 版本
  - [three-forcegraph](https://github.com/vasturiano/three-forcegraph) — Three.js 集成的 3D 力导向图
  - Fruchterman, T. & Reingold, E. "Graph Drawing by Force-directed Placement" (1991)
- **实施建议**: 在后端 Go 中实现力导向布局计算（新文章创建时重新平衡），结果存入 `physics_params.base_coordinates`。前端不做实时力模拟，仅渲染后端计算好的平衡位置

#### 7.3.3 t-SNE / UMAP 高维数据降维可视化

- **适用场景**: 将文章的高维特征（向量嵌入）映射到 3D 空间坐标，实现"语义宇宙"
- **当前方案**: 行星的 3D 坐标由博主手动设置或简单随机分配，空间位置与文章内容无关
- **先进替代**: **UMAP** (Uniform Manifold Approximation and Projection)——将文章的 768 维向量嵌入降维到 3 维坐标。语义相似的文章在 3D 空间中自然靠近，形成"意义星团"
- **视觉效果**: 宇宙的空间布局不再是随机的装饰，而是**信息的拓扑映射**——访客在宇宙中航行时，靠近的行星就是内容相关的文章。这让"探索宇宙"变成了一种直觉式的知识发现方式
- **参考资料**:
  - McInnes, L. et al. "UMAP: Uniform Manifold Approximation and Projection for Dimension Reduction" (2018) — [论文](https://arxiv.org/abs/1802.03426)
  - [umap-js](https://github.com/PAIR-code/umap-js) — Google 的 JavaScript UMAP 实现
  - [umap-learn (Python)](https://github.com/lmcinnes/umap) — 可在后端 Python 微服务中运行
- **实施建议**: 后端用 Python 微服务运行 UMAP（Go 调用），每次有新文章时重新计算全局降维，输出 (x, y, z) 坐标存入 DB。前端直接使用计算好的坐标渲染

#### 7.3.4 LLM 驱动的智能摘要与交互

- **适用场景**: 文章自动摘要（RSS/SEO）、搜索结果上下文、虫洞预览的年度总结
- **当前方案**: 摘要为前 300 字截断，缺乏语义概括
- **先进替代**: 使用本地部署的小型 LLM（如 Qwen2.5-7B 通过 Ollama）实现：
  - **文章发布时**自动生成 2-3 句摘要（存入 DB，供 RSS/SEO/虫洞预览使用）
  - **搜索时**对命中文章生成与查询词相关的上下文片段（而非机械高亮）
  - **年度虫洞预览**：自动生成"2024 年你写了 42 篇文章，主要关注 Go 后端和 3D 图形学..."的年度总结
- **参考资料**:
  - [Ollama](https://ollama.com/) — 本地 LLM 运行平台
  - [Qwen2.5](https://huggingface.co/Qwen/Qwen2.5-7B) — 中文能力优秀的开源模型
- **实施建议**: 作为异步后台任务运行（文章发布后 10 秒内完成），不阻塞用户操作。如服务器资源有限可用 API（如 Claude API）替代本地模型

### 7.4 下一代 Web 平台

#### 7.4.1 WebGPU Compute Shader (GPU 通用计算)

- **适用场景**: N-body 引力模拟（§7.2.1）、Boids 群集（§7.2.2）、Curl Noise 大规模粒子场、Reaction-Diffusion 实时迭代
- **当前方案**: 所有粒子逻辑在 CPU 的 `useFrame` 中计算，5000+ 粒子时成为瓶颈
- **先进替代**: **WebGPU Compute Shader**——直接在 GPU 上运行通用并行计算（非渲染），一个 Dispatch 可处理百万粒子的位置更新。比 WebGL GPGPU（将数据编码为纹理的 hack）更直接、更高效
- **现状**: Chrome 113+、Edge 113+、Firefox Nightly 已支持。Safari 部分支持。覆盖率约 70%（2025 年中）
- **参考资料**:
  - [WebGPU 官方示例: Compute Boids](https://webgpu.github.io/webgpu-samples/?sample=computeBoids) — 经典入门
  - [WebGPU Fundamentals](https://webgpufundamentals.org/) — 系统教程
  - [Three.js WebGPU 后端 (TSL)](https://threejs.org/docs/#manual/en/introduction/How-to-use-WebGPU) — Three.js r160+ 的 WebGPU 渲染器
  - [Use.GPU](https://usegpu.live/) — 声明式 WebGPU 框架
- **实施建议**: 作为**渐进增强**方案——检测到 WebGPU 支持时启用 GPU Compute 粒子系统，否则回退到 WebGL 的 GPGPU（数据纹理方案）或 CPU 计算。Three.js r160+ 已内置 WebGPU 后端，可通过 `WebGPURenderer` 替代 `WebGLRenderer`

#### 7.4.2 WebTransport (HTTP/3 低延迟通信)

- **适用场景**: 在线用户位置同步（PLAN_06 §11.4）、评论实时推送（PLAN_06 §11.2）
- **当前方案**: WebSocket（基于 TCP），连接建立需 1-RTT，受队头阻塞（HOL blocking）影响
- **先进替代**: **WebTransport**——基于 HTTP/3 (QUIC) 的新一代实时通信协议。支持不可靠数据报（Datagram，类似 UDP）和可靠流（Stream），建连延迟更低（0-RTT），无队头阻塞
- **优势对比**:
  | 特性 | WebSocket | WebTransport |
  |------|-----------|-------------|
  | 传输层 | TCP | QUIC (UDP) |
  | 建连延迟 | 1-2 RTT | 0-1 RTT |
  | 队头阻塞 | 有 | 无 |
  | 不可靠传输 | 不支持 | 支持 Datagram |
  | 适用场景 | 评论推送（需可靠） | 位置同步（可丢包） |
- **参考资料**:
  - [MDN: WebTransport API](https://developer.mozilla.org/en-US/docs/Web/API/WebTransport)
  - [WebTransport over HTTP/3 (RFC 9297)](https://www.rfc-editor.org/rfc/rfc9297)
  - Go 实现: [quic-go/webtransport-go](https://github.com/quic-go/webtransport-go)
- **实施建议**: 用户位置同步使用 WebTransport Datagram（丢包无所谓，下一包会覆盖），评论推送仍用 WebSocket 或 WebTransport Stream（需可靠）。检测不支持 WebTransport 时回退到 WebSocket

#### 7.4.3 View Transitions API (视图过渡动画)

- **适用场景**: 3D 场景与 2D 阅读面板的切换（大气突入 → 文章阅读 → 返回宇宙）、虫洞跃迁的页面级过渡
- **当前方案**: 手动编排 GSAP 动画 + DOM 切换 + WebGL 状态管理，代码复杂且容易出现闪烁
- **先进替代**: **View Transitions API**——浏览器原生提供的页面状态过渡动画 API。浏览器自动对"旧视图"和"新视图"做快照，在两个快照间执行交叉淡入/淡出/滑动等过渡，零闪烁
- **参考资料**:
  - [MDN: View Transitions API](https://developer.mozilla.org/en-US/docs/Web/API/View_Transitions_API)
  - [Chrome Blog: View Transitions](https://developer.chrome.com/docs/web-platform/view-transitions)
  - Next.js 支持: `next/navigation` 的 `startViewTransition` 集成（实验性）
- **实施建议**: 用于 3D → 2D DOM 面板的过渡（如文章阅读面板的出现/消失），替代部分手动 CSS/GSAP 过渡。3D 场景内部的动画仍由 GSAP 控制

### 7.5 安全前沿

#### 7.5.1 Argon2id (下一代密码哈希)

- **适用场景**: 博主登录密码哈希（PLAN_02 §2.5，当前使用 bcrypt）
- **先进替代**: **Argon2id**——2015 年密码哈希竞赛（PHC）冠军算法。同时抵抗 GPU 暴力破解（memory-hard）和侧信道攻击（data-independent memory access），安全性优于 bcrypt/scrypt
- **参考资料**:
  - [Go Argon2 标准库](https://pkg.go.dev/golang.org/x/crypto/argon2) — `golang.org/x/crypto/argon2`
  - [OWASP 密码存储推荐](https://cheatsheetseries.owasp.org/cheatsheets/Password_Storage_Cheat_Sheet.html) — 首选 Argon2id
- **实施建议**: 直接替换 bcrypt，Go 标准扩展库已内置。推荐参数：`time=3, memory=64MB, threads=4`

#### 7.5.2 Shamir's Secret Sharing (秘密分享方案)

> **注意**：加密文章功能已废弃（黑洞改为星系中心视觉元素），此算法当前无直接使用场景。保留作为未来可能需要保护敏感配置（如 API Key 备份）时的参考。

- **核心概念**: 将密钥拆分为 N 份，任意 K 份即可重建原始密钥
- **参考资料**:
  - Shamir, A. "How to Share a Secret" (Communications of the ACM, 1979)
  - [Go 实现: corvus-ch/shamir](https://github.com/corvus-ch/shamir)
